**작성일**: 2026-04-16
**작성자**: planner-architect
**대상 파일**: `illustrator-scripts/grading.jsx` (STEP 1~11, 약 2014줄)
**목적**: 사이즈별 요소가 아트보드를 수백 pt 벗어나는 회귀를 뽑아내기 위해 현재 설계의 가정을 해체하고, 5개 옵션을 비교·권장안을 제시한다.

## 0. 비유로 먼저 이해하기

현재 구조는 **"옷(몸판) 조각 위에 스티커(요소)를 붙이는 작업"** 같다.

- **기준(XL) AI**: 이미 스티커가 적절한 위치에 붙어 있는 옷 한 벌 → 정답지
- **타겟(예: 3XL) SVG**: 실루엣만 있고 스티커는 없는 더 큰 옷
- **스크립트의 목표**: 타겟 옷에 기준 스티커를 같은 의도로 다시 붙이기

그런데 지금 구현은 이런 전제를 깔고 있다:

> "**기준 옷과 타겟 옷의 조각 수가 같고 순서도 같다**.
> 각 조각의 스티커를 떼어, 조각 중심이 이동한 만큼 스티커 중심도 따라 이동시킨다."

→ 조각이 3XL에서 벌어지면(옷이 커지면 앞/뒤 조각 간격도 넓어짐) **스티커도 따라 벌어져서 아트보드 바깥으로 튀어나간다.** 이것이 보고된 증상의 수학적 원인이다.

## A. 현재 설계 요약 (STEP별 역할/가정/한계)

| STEP | 역할 | 핵심 가정 | 한계/취약점 |
|---|---|---|---|
| 2A | 디자인 AI "패턴선" 50pt+ path `area` 합 = **baseArea** | 패턴선 레이어 = 몸판 조각 경계선 | 조각 면적 ≠ 옷 사이즈 (곡선/너치가 노이즈) |
| 3~4 | "몸판" 레이어 색 / "요소" 레이어 copy + 각 요소의 `designPieces` 매칭 인덱스 사전 수집 | 요소 하나는 한 조각에 속함 (교집합 최대) | 요소가 2조각 걸치거나, 앞/뒤 쌍으로 배치되면 오매칭 |
| 4B | "교체용요소" 레이어 선택 탐색 | 아트보드 밖 백넘버는 위치 고정, 스케일만 | 교체용요소가 아트보드 안에 있거나 섞이면 분리 실패 |
| 5 | 타겟 SVG 열고 아트보드 크기로 CMYK 빈 문서 생성 | SVG 아트보드 ≈ 옷 실루엣 외접 박스 | 아트보드는 사이즈 무관 고정(4337×3401)일 수도 있음 (로그상 그렇다) |
| 7 | SVG 50pt+ path → 베이스 문서 복제 + `targetArea` 합산 + basePieces(bbox, cx, cy) 저장 X 오름차순 정렬 | 조각이 X축 순서로 일관 | 좌/우 대칭 조각이면 X 정렬 불안정 |
| 8 | clipboard(요소) → "디자인 요소" 레이어에 paste | paste 순서가 copy 순서와 동일 | Illustrator paste는 z-order/순서 보장 약함 |
| 8B | 교체용요소 copy/paste(pasteInPlace 우선) | 원본 좌표 유지 + selection 오염 방어 | selection 오염 버그 경험 있음 |
| 9 | 요소 전체 그룹 → `linearScale = sqrt(targetArea/baseArea)`로 **중심 기준** resize | 면적비가 옷 "크기"의 참값 | 면적은 곡선·너치·패턴선 선폭 영향을 누적 흡수 |
| 9B | 교체용요소 각 아이템 자기 중심 기준 scale (위치 유지) | 위치 이동 없음 | 문제 없음 (격리됨) |
| 10 | 그룹 해제 → 각 요소를 `basePiece.cx + relOffset × linearScale` 위치로 translate | **요소 상대 위치 = designPiece 중심 기준, scale에 비례** | ★★★ **핵심 취약점** (아래 B·C·D 참조) |
| 11 | 레이어 통합 → 저장 | — | — |

## B. 실측 데이터 (사용자 제공 로그 요약)

**기준**: XL, `baseArea = 7,468,498 pt²`

| 사이즈 | targetArea | areaRatio | linearScale | 요소 bounds (pt) | 아트보드 (4337×3401) 초과 |
|---|---:|---:|---:|---|---|
| 4XS | 4,968,288 | 0.665 | 0.8155 | 2951 × 2290 | 없음 |
| 3XS | 5,296,447 | 0.709 | 0.8422 | — | — |
| 2XS | 5,634,321 | 0.754 | 0.8685 | — | — |
| XS | 5,981,871 | 0.801 | 0.8949 | — | — |
| S | 6,338,981 | 0.849 | 0.9213 | 3411 × 3442 | 경계 근접 |
| M | 6,705,864 | 0.898 | 0.9476 | — | — |
| XL(기준) | 7,468,479 | 1.000 | 1.0000 | 3609 × **3715** | 세로 **+316** |
| 2XL | 7,864,293 | 1.053 | 1.0262 | — | — |
| 3XL | 8,269,741 | 1.107 | 1.0522 | 3953 × **4290** | 세로 **+889** (상 632 + 하 257) |
| 4XL | 8,684,849 | 1.163 | 1.0784 | 3992 × **4331** | 세로 **+930** |

**핵심 수치 검증**:

- `linearScale` XL→4XL 증분: +7.8%
- 요소 가로 증분: 3609 → 3992 = **+10.6%**
- 요소 세로 증분: 3715 → 4331 = **+16.6%**

→ **세로 증가폭이 scale보다 2배 이상 빠르다.** 원인: STEP 10이 요소별로 `basePiece.cy` 방향으로 끌어당기는데, 사이즈가 커질수록 조각 간 세로 거리가 더 크게 벌어짐. 그 벌어짐이 스케일 효과에 **추가로 누적**된다.

```
       (간단 도식)
     XL 조각 배치          3XL 조각 배치
     [A]  [B]  [C]         [A]     [B]     [C]    ← 조각 간격이 벌어짐
      ↑    ↑    ↑            ↑        ↑        ↑
      a    b    c            a'        b'        c'   ← 요소도 따라 벌어짐
                             ← (요소 세로 bounds 폭발) →
```

## C. 근본 원인 가설

### C1. [확정적] 조각 "간격"이 요소 bounds에 1:1로 전가됨
STEP 10 공식 `newCenter = basePiece.center + origRelOffset × linearScale`은 **"요소가 조각에 붙어다닌다"** 를 수학적으로 강제한다. basePiece 간격이 XL→3XL에서 Δ 벌어지면, 요소 간격도 Δ 만큼 벌어진다. (linearScale과 **독립**적으로 추가 발생)

### C2. [강함] `linearScale` 기준 선택이 불안정
`baseArea`는 디자인 AI "패턴선"의 path.area 합, `targetArea`는 SVG의 path.area 합 — **두 다른 원천**. 패턴선 선폭·너치·곡선 근사 차이로 면적이 오차 포함. XL에서 `targetArea/baseArea = 0.9998`(거의 같음)은 "우연"이 아니라 둘 다 같은 패턴 생성 툴의 결과라서 그런 것. 3XL은 SVG만 독립적으로 scale-up 되어 **면적비 증가율이 실제 길이비보다 큼** (면적은 길이의 제곱이므로 노이즈도 제곱으로 증폭).

### C3. [강함] 기준점(원점/중심) 혼용
- `pastedGroup.resize(scale, scale, ..., true, true, true, true)` — 여덟 번째 인자가 없으면 기본 `Transformation.DOCUMENTORIGIN` (0,0 기준). 그룹 bounds 중심이 (1800, -1500)쯤인데 원점(0, svgHeight)에서 스케일 → **스케일 즉시 위치가 크게 이동** 한다.
- 코드 주석은 "중심 기준"이라 썼지만 실제 API 기본값은 origin. STEP 10이 뒤이어 조각 중심으로 재이동하므로 겉으론 괜찮아 보이지만, 요소가 멀리 날아갔다 돌아오는 과정에서 누적 오차가 커질 수 있다.
- STEP 9B 교체용요소는 명시적으로 `Transformation.CENTER` 지정 → **대칭 불일치**.

### C4. [중간] 조각 매칭 불안정
X 오름차순 정렬은 좌/우 소매가 같은 X 중심을 가질 때 불안정. 사용자 승인 규칙 "디자인 AI 조각과 SVG 조각 순서 동일"이 특정 패턴에서 깨지면 요소가 엉뚱한 조각에 매핑된다. 사이즈 커질수록 오매칭의 위치 오차도 커짐.

### C5. [약함] SVG 아트보드와 옷 실루엣 불일치
SVG의 `viewBox`가 모든 사이즈에서 동일(4337×3401) — 즉 **아트보드는 사이즈와 무관**. 옷 크기는 내부 path들의 bbox 합으로만 표현. `baseDoc`을 SVG 아트보드로 만들면 모든 사이즈 베이스가 같은 크기 → 옷이 커지면 자연스럽게 바깥으로 삐져나갈 여지.

## D. 재설계 옵션 5가지

### D1. 옵션 1 — "요소는 몸판 전체 중심 고정, 스케일만" (가장 단순)

**발상**: 요소 전체를 한 덩어리로 스케일 후, 몸판 전체 bbox 중심으로 딱 한 번 이동. STEP 10(조각별 정렬) 전면 제거.

```
구현 규모: STEP 10 블록 삭제 + alignToBodyCenter 호출로 단일화 (~-120줄)
난이도: ★☆☆☆☆
리스크: ★☆☆☆☆ (이미 폴백 경로로 존재)
```

**장점**:
- 아트보드 초과 원천 차단 (C1 해소)
- 디버깅/예측 쉬움 (하나의 중심 + 하나의 스케일)
- 구현 1시간 이내 완료

**단점**:
- 사이즈별로 몸판 조각 간격이 벌어져도 요소는 몸판 중심에 뭉쳐있음
- 앞판 로고 / 뒷판 넘버를 **독립적으로** 움직여야 하는 경우 지원 불가

**적합 조건**:
- 디자인이 "옷 중앙에 큰 하나"(예: 로고 단일) 위주일 때
- 앞/뒷판이 같은 아트보드에 연달아 배치되지 않고 별도 발주되는 워크플로우

### D2. 옵션 2 — "현재 로직 + 아트보드 클립 마스크"

**발상**: STEP 10 유지, 요소 전체 bounds를 아트보드 경계로 클립.

```
구현 규모: STEP 11 직전 클립 마스크 추가 (~+30줄)
난이도: ★★☆☆☆
리스크: ★★★☆☆ (요소가 잘려 출력물 품질 저하 — 문제 은폐)
```

**장점**: 기존 배치 철학 유지.
**단점**: **증상만 가림**. 잘린 요소는 사라져 출력 품질이 더 나빠진다. 추천 ✗.

### D3. 옵션 3 — "몸판 중심 상대 비율 배치" (균형형)

**발상**:
1. 기준 AI에서 요소 중심 위치를 **몸판 전체 bbox 대비 비율**(0~1)로 저장.
   `(relX, relY) = (itemCx - bodyMinX) / (bodyMaxX - bodyMinX), ...`
2. 타겟 SVG의 몸판 bbox를 구한다.
3. 요소를 `linearScale`로 스케일 후, **각 요소 중심을 타겟 몸판의 같은 비율 위치**로 translate.

```
구현 규모: STEP 10 재작성 (~-50줄/+70줄)
난이도: ★★★☆☆
리스크: ★★☆☆☆
```

**장점**:
- 몸판이 커지면 요소도 **몸판 안에서** 균형 있게 분산 (조각 간격은 따라가지만 기준이 "몸판 전체")
- C1(조각 간격 전가) 완화 — 간격이 몸판 bbox 전체로 정규화되므로 폭발적 증가 제거
- 아트보드 초과 자동 방지 (요소가 몸판 bbox 안쪽에 배치)

**단점**:
- 요소가 특정 조각(소매만 등) 위에 의도된 경우 부정확
- 비율 기준점이 bbox 모서리라 대칭 어긋나면 치우침

**핵심 수식**:
```
// 기록 (기준 AI, 한 번만)
for each item_i:
  relX_i = (item_i.cx - baseBodyBbox.minX) / baseBodyBbox.width
  relY_i = (item_i.cy - baseBodyBbox.minY) / baseBodyBbox.height

// 배치 (타겟 SVG)
for each paste_item_i:
  targetCx = targetBodyBbox.minX + relX_i * targetBodyBbox.width
  targetCy = targetBodyBbox.minY + relY_i * targetBodyBbox.height
  translate to (targetCx, targetCy)
```

### D4. 옵션 4 — "앞/뒤 조각별 요소 그룹 사전 분리 (디자이너 협업 필요)"

**발상**: 기준 AI에 레이어를 `요소-앞판`, `요소-뒷판`, `요소-소매` 등으로 분리. 각 그룹이 해당 조각과 1:1 매핑. 스크립트는 그룹별로 독립 스케일/배치.

```
구현 규모: STEP 3~10 전반 재작성 + 디자이너 워크플로우 변경 (~+200줄)
난이도: ★★★★☆
리스크: ★★★★☆ (기존 디자인 AI 전수 재작업 필요)
```

**장점**:
- 명시적 · 가장 정확
- 각 조각 독립 스케일 가능 (부분별 비율 다름에 대응)

**단점**:
- **기존 디자인 AI 수십~수백 개를 레이어 분리 작업** 해야 함 (디자이너 공수 큼)
- 구현도 복잡 (레이어 네이밍 규칙 정립, 누락 시 폴백 등)

**적합 조건**: MVP 이후 장기 로드맵.

### D5. 옵션 5 — "기준점 완전 고정 (아트보드 중심)"

**발상**: 요소 전체 bbox 중심을 **아트보드 중심** 에 강제 고정. 스케일 외 어떤 보정도 없음. (옵션 1의 변형 — 몸판 대신 아트보드)

```
구현 규모: 옵션 1과 동일 (~-120줄)
난이도: ★☆☆☆☆
리스크: ★★☆☆☆ (몸판이 아트보드에서 오프셋된 경우 요소와 몸판이 어긋남)
```

**장점**: 가장 예측 가능. 디버깅 극단적으로 쉬움.
**단점**: 몸판 자체가 아트보드 중앙에 없으면 요소와 몸판이 정합 안 됨.

## F. 권장안

### 🏆 1순위 — **D1 (몸판 전체 중심 고정, 스케일만)**

**이유** (바이브 코더 친화):
1. **지금 이미 "폴백 경로"로 돌아가는 로직이 있다.** STEP 10의 `useFallback` 분기에서 `alignToBodyCenter` 호출. 이걸 "정상 경로로 승격"시키는 작업만 하면 된다.
2. 코드 변경 최소 (한 블록 삭제 + 몇 줄 정리)
3. 옷 중심에 로고 하나인 단순 디자인(현재 대부분)에 충분
4. 실패 시 **2순위 D3로 확장 가능** — D1 먼저 현업 검증 후 부족하면 D3 추가

### 🥈 2순위 — **D3 (몸판 bbox 비율 배치)**

**D1이 불충분할 때 전환**. 요소가 3개 이상이고 앞/뒤 분산 디자인이 많다면 D3가 맞다.

### ❌ 권장 안 함 — **D2 (클립), D4 (그룹 분리)**

- D2는 문제 은폐
- D4는 기존 디자인 AI 전수 재작업 필요 → MVP 범위 외

## H. 사용자 의사결정 체크리스트

답변이 필요한 항목 (A/B/C 선택):

### Q1. 사이즈가 커지면 요소 간격은?
- **A.** 고정 — 요소는 몸판 중앙에 뭉쳐있고, 몸판만 벌어짐 (D1 선택)
- **B.** 같이 벌어짐 — 몸판 조각이 벌어지는 만큼 요소도 비례로 벌어짐 (D3 선택)
- **C.** 앞/뒤 요소가 완전 독립 — 각 조각에 고정된 요소 (D4 선택, 레이어 재작업 필요)

### Q2. 기준 AI의 "요소" 레이어에 몇 종류 요소가?
- 예: [로고 1개] / [로고 + 번호 2개] / [로고 + 번호 + 이름 3개]
- 숫자만 알려줘도 됨. **2개 이하면 D1로 충분, 3개 이상이면 D3 권장.**

### Q3. 앞판/뒷판이 같은 아트보드에 함께 있는가?
- **A.** 예 (앞뒤가 좌우로 나란히) → D1/D3 모두 가능
- **B.** 아니오 (별도 발주) → D1로 간소화 가능

### Q4. 요소가 "조각 경계를 가로지르는" 디자인이 있는가?
- 예: 어깨~가슴 걸쳐 있는 큰 로고
- **A.** 예 → D1/D3 (그룹 전체 처리)
- **B.** 아니오 → D4도 고려 가능

### Q5. D4(디자이너 레이어 분리) 수용 가능?
- 기존 디자인 AI 전수 재작업 필요 (수십~수백 개)
- **A.** 불가 → D1/D3 중 선택
- **B.** 가능 (새 디자인부터 적용) → 장기 로드맵으로 기록
