# 의류 그레이딩 지침 적용 가능성 검토 보고서

## 요약

현재 grading.jsx(v2)는 "디자인 AI의 상대 위치를 SVG에 재현"하는 방식으로, 패턴의 기준선/시접 같은 봉제 규격 개념이 전혀 없다. 사내 지침의 핵심은 "밑단 기준선에서 5cm 위" 같은 **절대 오프셋 배치**인데, 현재 코드는 패턴에서 "밑단이 어디인지"를 식별할 수 없다. 대지 사이즈 변경(158x200cm)은 쉽게 적용 가능하나, 기준선 인식/시접 계산은 **SVG에 메타데이터를 추가하는 인프라 작업이 선행**되어야 한다. 단기적으로 config.json에 수동 오프셋 값을 넣어 "반자동" 적용하는 중간 단계가 현실적이다.

## 1. 항목별 분석 매트릭스

| # | 지침 항목 | 현재 상태 | 적용 가능성 | 난이도 | 예상 공수 | 비고 |
|---|---------|---------|-----------|-------|---------|------|
| 1 | 대지 사이즈 158x200cm | 미구현. SVG viewBox 크기 사용 | 추가 구현 가능 | 쉬움 | 0.5일 | 158cm=4478.7pt, 200cm=5669.3pt |
| 2 | 상의 옆면 시접 0.7cm | 미구현. 시접 개념 없음 | 조건부 가능 | 어려움 | 3~5일 | "옆선" 식별 필요 |
| 3 | 상의 밑단 시접 2cm | 미구현 | 조건부 가능 | 어려움 | 3~5일 | "밑단" 식별 필요 |
| 4 | 뒷목 로고: 기준선에서 3.5cm 아래 | 미구현 | 조건부 가능 | 중간 | 2~3일 | "뒷목 기준선" + 요소 유형 구분 필요 |
| 5 | 사이즈택: 밑단에서 5cm 위 | 미구현 | 조건부 가능 | 중간 | 2~3일 | "밑단 기준선" + 요소 유형 구분 필요 |
| 6 | 그래픽 시접 침범 금지 | 미구현 | 조건부 가능 | 어려움 | 5~7일 | 시접 경계 + 충돌 감지 필요 |
| 7 | 허리밴드 10.5cm | 미구현 | 조건부 가능 | 중간 | 2일 | 검증/조정 로직 추가 |
| 8 | 하의 밑단 시접 2cm | 미구현 | 조건부 가능 | 어려움 | 3~5일 | 항목 3과 동일 |
| 9 | 하의 이니셜/로고: 밑단에서 5cm 위 | 미구현 | 조건부 가능 | 중간 | 2~3일 | 항목 5와 동일 |
| 10 | 상의/하의 구분 | 미구현 | 추가 구현 가능 | 중간 | 1~2일 | config에 garmentType 추가 |

> **"조건부 가능"**: 코드 구현은 가능하지만, 패턴에서 기준선을 식별하는 인프라가 선행 필요.

## 2. 핵심 기술 과제

### 2-1. 기준선 자동 식별 문제

비유: 지금 코드는 "종이 위의 도형 크기만 재는 자"다. 지침이 요구하는 건 "이 도형의 목선은 여기, 밑단은 저기"를 알아내는 것인데, 종이에 "여기가 목선입니다"라고 적혀 있지 않으면 자만으로는 알 수 없다.

가능한 접근법:

| 접근 | 설명 | 난이도 | 정확도 |
|------|------|--------|--------|
| A. config 수동 입력 | 기준선 y좌표를 config.json에 직접 입력 | 쉬움 | 높음 |
| B. bbox 휴리스틱 | bbox top=목선, bottom=밑단 가정 | 중간 | 중~낮 |
| C. SVG 이름 규약 | path에 id="neckline" 등 태그 | 중간 | 높음 |
| D. 마커 레이어 | SVG에 "기준선" 레이어 추가 | 중간 | 높음 |
| E. 경로 분석 | anchor point로 직선/곡선 자동 판별 | 매우 어려움 | 중 |

권고: 단기 A(config 수동) → 중기 C/D(SVG 이름/마커)

### 2-2. 시접 영역 계산 문제

시접은 "봉제할 때 접어 넣는 여분의 천". 그래픽은 시접 안쪽(보이는 영역)에만 있어야 한다. 단순화: 밑단/옆면이 직선이면 bbox 기준 오프셋 가능 (밑단 +56.7pt, 옆면 +19.8pt).

### 2-3. 상의/하의 자동 구분 문제

자동 구분은 예외가 많아 불안정. config.json에 garmentType 필드 추가가 확실하고 쉽다.

### 2-4. 절대 위치 vs 상대 위치 충돌

현재 B-2안: `요소 위치 = SVG 조각 중심 + 상대벡터 x 스케일`. 지침: `뒷목 로고 = 기준선 - 3.5cm` (절대 오프셋). 두 방식은 공존 가능 -- 이름이 지정된 요소(뒷목로고, 사이즈택)는 절대 배치, 나머지는 상대 배치 유지.

## 3. 필요한 추가 인프라

### 3-1. 패턴 메타데이터

| 데이터 | 필수 여부 | 제공 방법 |
|--------|-----------|-----------|
| 기준선 y좌표 | 필수 | config.json 수동 또는 SVG 마커 |
| 시접 수치 | 필수 | config.json (기본값 하드코딩 가능) |
| 의류 종류 | 필수 | config.json garmentType |
| 허리밴드 높이 | 선택 | config.json 또는 기본값 10.5cm |

### 3-2. 디자인 AI 레이어 규약 추가

- 요소 아이템 이름 지정 ("뒷목로고", "사이즈택", "그래픽_01")
- 선택: "기준선" 레이어 추가 (목선/밑단을 명시적 path로 표기)

### 3-3. config 확장 제안

```json
{
  "garmentType": "top",
  "guidelines": {
    "enabled": true,
    "artboardCm": [158, 200],
    "seam": { "side": 0.7, "hem": 2.0 },
    "placement": { "backNeckLogo": 3.5, "sizeTag": 5.0, "bottomLogo": 5.0 },
    "waistband": 10.5
  },
  "referenceLines": { "necklineY": null, "hemlineY": null }
}
```

## 4. 구현 우선순위

| 순위 | 항목 | 이유 | 의존 관계 |
|------|------|------|---------|
| 1 | 대지 사이즈 158x200cm | 코드 1곳 수정으로 즉시 적용 | 없음 |
| 2 | config garmentType + 분기 뼈대 | 모든 지침의 전제 조건 | 없음 |
| 3 | config 시접 수치 + 기본값 | 하드코딩 가능 | 순위 2 |
| 4 | bbox 기반 시접 침범 경고 | 로그 출력만 | 순위 3 |
| 5 | 요소 이름 유형 구분 + 절대 배치 | 핵심. AI 이름 규약 선행 필요 | 순위 2 + 이름 규약 |
| 6 | 시접 침범 자동 보정 | 안전 영역으로 밀어넣기 | 순위 4 |
| 7 | SVG 마커 기준선 자동 인식 | 완전 자동화 최종 단계 | export 파이프라인 |

## 5. 한계점 및 리스크

### 5-1. 자동화 불가능한 영역
- 곡선 시접 계산 (path offset 프로그래밍 제어 제한적)
- 봉제 후 위치 어긋남 예측 (물리적 요인, SW 자동화 불가)
- 래글런/요크/플레어 등 특수 패턴 (bbox 가정 깨짐)

### 5-2. 기존 로직 충돌
- B-2안 상대 배치 vs 지침 절대 배치: 요소 유형 분기로 해결 가능
- 균일 스케일 vs 고정 오프셋: 스케일 후 절대 위치 재조정으로 해결
- 대지 크기 변경 시 패턴이 좌상단 몰림 가능: 중앙 배치 보정 로직 필요

### 5-3. 정밀도 리스크
- cm→pt 변환 상수(28.3465) 미정의 -- 추가 필요
- bbox 기반 기준선은 V넥에서 수 cm 오차 가능
- SVG viewBox(4337x3401)와 실제 cm 비율 불명 -- 패턴 SW export 설정 확인 필요

## 6. 권고 사항

**단기 (1~2주)**: config 확장 + 대지 사이즈 고정 + 시접 경고 로그 + CM_TO_PT 상수. "반자동" 수준.

**중기 (1~2개월)**: 디자인 AI 이름 규약 도입 + 요소 유형별 절대/상대 배치 분기 + bbox 기준선 추정 + 시접 보정.

**장기 (3개월+)**: SVG 마커 레이어 + 정밀 기준선 인식 + 곡선 시접 + 좌표-cm 자동 매핑. 완전 자동화.

## 7. 결론

10개 지침 중 **즉시 적용 1개**, **config 확장 반자동 4개**, **인프라 선행 필요 5개**. 가장 큰 장벽은 "패턴에서 기준선을 어떻게 식별하는가"이며, 이것은 코드 문제가 아니라 **입력 데이터에 의미 정보가 없는 문제**이다. 코드 수정보다 **디자인 파일 제작 규약을 먼저 확정**하는 것이 올바른 순서다.
