# 패턴별 요소 재구성 방식 - 타당성 검토 및 구현 계획

> 작성일: 2026-04-08 | 작성자: planner-architect
> 대상: 승화전사 유니폼 패턴 자동 생성 프로그램 (grader)

---

## 1. 이 보고서가 다루는 것

### 사용자 제안 요약

> "디자인을 통째로 축소하는 게 아니라, 요소별로 분해해서 재구성하자"

현재 방식(grading.jsx)은 **디자인 전체를 패턴 틀 안에 넣고 잘라내는** 방식이다.
사용자 제안은 디자인의 **개별 요소(배경색, 로고, 번호, 스트라이프 등)를 따로 추출**해서
타겟 패턴 위에 **하나씩 배치**하는 방식이다.

### 비유

- **현재 방식** = 큰 사진을 쿠키 커터로 찍어내기 (사진 전체를 틀 안에 밀어넣고 자름)
- **제안 방식** = 빈 쿠키 반죽에 토핑을 하나씩 올리기 (틀에 먼저 바탕색을 칠하고, 로고/글자를 따로 올림)

실제 디자이너가 수작업으로 그레이딩할 때는 **후자(토핑 방식)**에 가깝다.

## 2. 실제 디자이너 수작업 과정

디자이너가 XL 디자인을 XS로 그레이딩할 때 하는 일:

```
1단계: XS 패턴 SVG를 새 아트보드에 배치          <- "빈 틀 준비"
2단계: XL 디자인에서 배경색(메인 컬러) 추출       <- "바탕색 확인"
3단계: XS 패턴 조각 안에 배경색 채우기             <- "틀에 색칠"
4단계: XL 디자인에서 로고, 텍스트, 번호 추출       <- "토핑 준비"
5단계: 각 요소를 XS 패턴에 맞는 위치에 배치        <- "토핑 올리기"
6단계: 미세 조정 (정렬, 간격)                     <- "마무리"
7단계: PDF로 내보내기                             <- "완성"
```

핵심: 디자이너는 디자인을 **통째로 축소하지 않는다**. 요소를 **분해해서 재조립**한다.

## 3. 현재 방식 vs 제안 방식 비교

| 항목 | 현재 (통째로 클리핑) | 제안 (요소 재구성) |
|------|---------------------|-------------------|
| 비유 | 큰 사진을 틀로 자르기 | 빈 틀에 토핑 올리기 |
| 배경색 처리 | 디자인 전체가 들어가므로 자연스럽게 채워짐 | 패턴 형태에 맞게 정확히 채워짐 |
| 로고 위치 | 디자인 원본 위치 그대로 (패턴 밖이면 잘림) | 패턴 안에 정확히 배치 가능 |
| 텍스트/번호 | 비례 축소되어 위치가 밀릴 수 있음 | 원하는 위치에 정밀 배치 |
| 구현 난이도 | 낮음 (현재 완성됨) | 매우 높음 (새로 개발 필요) |
| 사전 조건 | 없음 | 디자인 파일의 레이어 구조 정형화 필요 |
| 디자이너 부담 | 없음 | 작업 규칙(어떤 요소가 어디에) 정의 필요 |
| 결과 품질 | 대부분 OK, 일부 부정확 | 디자이너 수작업과 동일 품질 |

## 4. 핵심 기술적 질문과 답변

### Q1. 디자인 파일의 요소를 자동으로 분류할 수 있는가?

**답: 자동 분류는 현실적으로 불가능하다.**

디자인 파일 안의 요소들:
- 배경색 (단색, 그라데이션, 패턴 채우기)
- 스트라이프 (줄무늬, 옆선)
- 로고 (팀 엠블럼, 브랜드)
- 텍스트 (팀명, 선수명, 번호)
- 장식 (리본, 테두리)

이것들을 프로그램이 "이건 로고고, 이건 배경이다"라고 자동 판별하는 건
AI/머신러닝 수준의 기술이 필요하다. 우리 프로젝트 범위를 벗어난다.

**대안: 레이어 이름 규칙으로 구분**

디자이너가 Illustrator에서 디자인할 때 레이어 이름을 규칙에 맞게 정해놓으면,
프로그램이 이름을 보고 "배경 레이어구나, 로고 레이어구나" 판별할 수 있다.

```
비유: 서류함에 라벨을 붙여놓으면 누구든 찾을 수 있는 것과 같다.
      라벨이 없으면 서류를 하나씩 열어봐야 하지만,
      "계약서", "영수증" 이렇게 적어놓으면 바로 찾을 수 있다.
```

### Q2. Illustrator ExtendScript로 레이어별 요소 접근이 가능한가?

**답: 완벽하게 가능하다.**

Illustrator ExtendScript는 다음을 모두 지원한다:
- 레이어 이름으로 접근 (예: doc.layers["배경"])
- 그룹 안의 개별 아이템 접근
- 채우기 색상 읽기/쓰기 (단색, 그라데이션, 패턴 모두)
- 위치/크기 읽기/쓰기
- 복사/붙여넣기

현재 grading.jsx에서도 이미 pathItems, groupItems, position, resize 등을 사용 중이다.
**기술적 장벽은 없다.**

### Q3. 패턴 경로 안에 색상 채우기가 가능한가?

**답: 가능하다. 단색/그라데이션/패턴 채우기 모두 지원.**

- 단색: pathItem.fillColor = new CMYKColor()
- 그라데이션: pathItem.fillColor = new GradientColor()
- 패턴 채우기: pathItem.fillColor = new PatternColor()

핵심: 기준 디자인에서 채우기 색상을 **읽어서** 타겟 패턴에 **적용**할 수 있다.
그라데이션의 각도, 색상 정지점, 비율까지 모두 복사 가능.

### Q4. "작업 규칙"을 어떻게 정의하는가?

디자이너가 미리 정의해야 하는 규칙을 JSON 파일로 만든다:

```
비유: 요리 레시피 같은 것이다.
      "1번: 바탕은 빨간색으로 칠한다"
      "2번: 로고는 가슴 중앙에 놓는다"
      "3번: 번호는 등 중앙에 놓는다"
      이런 레시피를 프로그램에 알려주는 것.
```

각 요소에 대해:
- **이름**: 어떤 요소인지 (배경, 로고, 번호 등)
- **출처**: 기준 디자인의 어떤 레이어에서 가져오는지
- **규칙**: 타겟 패턴에서 어떻게 배치하는지 (중앙, 상단, 비례 축소 등)
- **우선순위**: 어떤 순서로 처리하는지 (배경 먼저, 로고 나중)

## 5. 자동화 가능 범위 (3단계)

### Level 1: 패턴 채우기만 자동화 (현실적, 즉시 착수 가능)

```
자동화 범위:
  [V] 패턴 SVG를 틀로 사용
  [V] 기준 디자인의 배경색/그라데이션 추출
  [V] 타겟 패턴 조각에 배경색 채우기
  [ ] 로고, 텍스트, 번호 배치 -> 디자이너가 수동

구현 기간: 1~2주
```

이것만으로도 디자이너 수작업의 **가장 시간이 많이 걸리는 부분**(색상 채우기)을 자동화한다.
나머지 요소(로고, 번호)는 디자이너가 직접 배치한다.

**비유: 자동으로 빵에 잼을 발라주고, 토핑은 사람이 올리는 것.**

### Level 2: 주요 요소 자동 배치 (중기, 작업 규칙 기반)

```
자동화 범위:
  [V] Level 1 전체
  [V] 로고를 지정 위치에 자동 배치
  [V] 번호/텍스트를 지정 위치에 자동 배치
  [V] 스트라이프를 패턴 크기에 맞게 스케일
  [ ] 복잡한 장식 요소 -> 디자이너 수동

사전 조건:
  - 디자인 파일의 레이어 구조 표준화
  - 작업 규칙 JSON 정의

구현 기간: 3~4주
```

### Level 3: 완전 자동화 (장기, AI 지원)

```
자동화 범위:
  [V] Level 2 전체
  [V] 요소 자동 인식 (레이어 이름 불필요)
  [V] 배치 위치 자동 결정
  [V] 디자인 패턴별 커스텀 규칙 학습

사전 조건:
  - AI/ML 모델 학습 데이터 (수십 개 디자인)

구현 기간: 2~3개월+
```

### 3단계 비교표

| 항목 | Level 1 | Level 2 | Level 3 |
|------|---------|---------|---------|
| 디자이너 작업량 감소 | 60% | 85% | 95% |
| 구현 기간 | 1~2주 | 3~4주 | 2~3개월+ |
| 사전 설정 | 없음 | 레이어 규칙 + JSON | AI 학습 데이터 |
| 기술 난이도 | 중 | 높음 | 매우 높음 |
| 추천 시기 | 지금 바로 | Level 1 검증 후 | 필요성 확인 후 |

## 6. 현재 방식과의 관계

**현재 grading.jsx는 폐기하는 게 아니라, Level 1의 기반이 된다.**

현재 grading.jsx의 흐름:
```
패턴 SVG 열기 -> 디자인 PDF Place -> 크기 맞추기 -> 클리핑 마스크 -> 저장
```

Level 1의 흐름:
```
패턴 SVG 열기 -> 기준 디자인에서 배경색 추출 -> 패턴에 색상 채우기
-> 기준 디자인을 Place -> 클리핑 마스크 -> 저장
```

차이점: 클리핑 마스크 **전에** 패턴 조각에 배경색을 먼저 채운다.
이렇게 하면 패턴 경계에서 색상이 빠지는 문제가 해결된다.

## 7. Level 1 상세 구현 계획

### 수정 대상: grading.jsx

현재 grading.jsx에 **3개 단계를 추가**하면 된다:

```
기존 STEP 1: 패턴 SVG 열기                    <- 유지
[신규] STEP 2A: 기준 디자인 PDF를 별도로 열기   <- 추가
[신규] STEP 2B: 배경 레이어의 색상 정보 추출    <- 추가
[신규] STEP 2C: 패턴 조각에 추출한 색상 채우기  <- 추가
기존 STEP 3: 디자인 PDF를 Place               <- 유지
기존 STEP 4~9: 크기 맞추기, 클리핑 등          <- 유지
```

### config.json 확장

새 필드:
- fillBackground: 배경색 채우기 활성화 (true/false)
- backgroundSource: "auto"(자동 감지) 또는 레이어명 지정

### 배경색 추출 방법

기준 디자인 PDF를 Illustrator로 열면, 가장 큰 면적을 차지하는 요소의
fillColor가 배경색일 가능성이 높다. 자동 감지 로직:

```
1. 디자인의 모든 pathItem을 순회
2. 각 pathItem의 면적(width x height) 계산
3. 가장 큰 면적의 pathItem의 fillColor를 배경색으로 결정
4. 그라데이션이면 GradientColor 객체 전체를 복사
```

### 리스크와 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 배경색 자동 감지 실패 | 잘못된 색상 채우기 | 수동 지정 모드 제공 (레이어명) |
| 그라데이션 각도 틀어짐 | 색상 방향 부정확 | 패턴 bbox 기준으로 각도 재계산 |
| 복잡한 패턴 채우기 | 단순 단색/그라데이션이 아닌 경우 | Level 1에서는 단색+그라데이션만 지원 |
| 디자인 PDF 구조 편차 | 디자이너마다 파일 구조가 다름 | 폴백: 현재 방식(클리핑만) 유지 |

## 8. Level 2 로드맵 (참고용)

Level 1이 안정화된 후 진행할 사항:

```
1. 디자인 파일 레이어 명명 규칙 확정
   - 예: "BG_배경", "LOGO_팀로고", "NUM_번호", "TEXT_팀명"
   
2. 작업 규칙 에디터 UI 추가
   - 사용자가 "로고는 가슴 중앙, 번호는 등 중앙" 같은 규칙을 UI에서 설정
   
3. 요소별 복사/배치 로직 구현
   - 기준 디자인에서 그룹 단위로 복사
   - 타겟 패턴의 지정 위치에 붙여넣기
   - 비율 조정 (패턴 크기 대비)

4. 규칙 템플릿 시스템
   - 자주 쓰는 규칙을 저장/불러오기
   - "축구 유니폼 기본", "농구 유니폼 기본" 등
```

## 9. 추천 방향

### 결론: Level 1부터 시작하되, 현재 방식을 폐기하지 않는다

1. **현재 grading.jsx는 그대로 유지** -- 이미 동작하고, 대부분의 경우 충분히 좋은 결과를 낸다.

2. **Level 1(배경색 채우기)을 grading.jsx에 추가** -- 기존 코드 위에 배경색 추출/채우기 단계만 끼워넣는다. 실패 시 기존 방식으로 폴백. 구현 기간 1~2주.

3. **Level 2는 Level 1 검증 후 결정** -- 실제 디자인 파일로 테스트해보고, 배경색 채우기만으로 충분한지 확인한 뒤 결정한다. 레이어 구조 표준화가 선행되어야 하므로 디자이너와 협의가 필요하다.

4. **Level 3는 현 단계에서 불필요** -- 현재 주문 규모(디자인 종류)에서는 과잉 투자.

### 왜 Level 1부터인가?

```
투자 대비 효과:
  Level 1: 1~2주 투자 -> 60% 수작업 감소    <- 가성비 최고
  Level 2: 3~4주 투자 -> 85% 수작업 감소    <- 효과 좋지만 사전 조건 많음
  Level 3: 2~3개월 투자 -> 95% 수작업 감소  <- 과잉 투자
```

**비유: 식기세척기를 사려고 주방 리모델링까지 할 필요 없다. 먼저 식기세척기만 사서 써보자.**

## 10. 실행 계획 (Level 1 구현)

| 순서 | 작업 | 담당 | 기간 | 선행 조건 |
|------|------|------|------|----------|
| 1 | grading.jsx에 배경색 추출 함수 추가 | developer | 2~3일 | 없음 |
| 2 | 패턴 조각 fillColor 적용 로직 추가 | developer | 2~3일 | 1 |
| 3 | config.json에 fillBackground 옵션 추가 | developer | 0.5일 | 2 |
| 4 | Rust/프론트에서 새 config 필드 전달 | developer | 1일 | 3 |
| 5 | 실제 디자인 파일로 테스트 | tester | 2~3일 | 4 |
| 6 | 폴백 로직 (실패 시 기존 방식) | developer | 1일 | 5 |

총 예상 기간: **1~2주**
