# 패턴 클리핑 마스크 기반 그레이딩 - 기술 타당성 보고서

> 작성일: 2026-04-08 | 작성: planner-architect

## 1. 왜 클리핑 마스크가 필요한가

### 현재 방식의 한계

현재 그레이딩은 **단순 비례 스케일링**이다. 기준 디자인(예: XL)을 가로/세로 비율로 줄이거나 늘려서 타겟 사이즈(예: XS)를 만든다.

하지만 실제 의류 패턴은 사이즈마다 **부위별로 다른 비율**로 변한다:
- 가슴 폭은 XS→XL에서 +10%인데, 길이는 +5%일 수 있음
- 목선 곡률, 암홀 커브, 어깨 경사 등 **형태 자체가 다름**

따라서 디자인을 단순 비율로 축소해도 실제 패턴 형태와 정확히 일치하지 않는다.

### 클리핑 마스크란?

**비유: 쿠키 커터**
- 반죽(= 스케일된 디자인)을 쿠키 커터(= 패턴 SVG)로 찍어내면, 틀 안쪽만 남고 바깥은 잘려나간다.
- Illustrator의 "클리핑 마스크" 기능과 동일한 개념이다.

현재 방식이 "사진을 축소해서 옷본에 올려놓기"라면, 클리핑 방식은 "사진을 축소한 후 옷본 모양으로 찍어내기"다.

---

## 2. 올바른 워크플로우

```
입력: 기준 디자인 PDF (XL) + XL 패턴 SVG + XS 패턴 SVG

Step 1. XL/XS 패턴의 대략적 크기 비율 계산 (현재 방식 그대로)
Step 2. 디자인 PDF를 해당 비율로 축소 (현재 방식 그대로)
Step 3. XS 패턴 SVG의 윤곽선 경로를 추출
Step 4. 축소된 디자인에 윤곽선을 클리핑 마스크로 적용
Step 5. 패턴 형태 안의 디자인만 남은 최종 PDF 출력
```

핵심: Step 1~2는 기존 코드를 그대로 쓰고, **Step 3~4만 추가**하면 된다.

## 3. 구현 방법 5가지 분석

### 방법 A: PyMuPDF 클리핑 패스

SVG 윤곽선 좌표를 PDF 클리핑 패스(경로)로 변환하여 PyMuPDF로 삽입한다.
- **장점**: 순수 벡터, CMYK 완벽 보존, 현재 기술 스택(PyMuPDF) 그대로 사용
- **단점**: SVG Bezier 곡선 -> PDF 클리핑 경로 변환 로직 직접 구현 필요
- **비유**: 빵틀(SVG)의 설계도를 읽어서, 같은 모양의 빵틀을 PDF 세계에서 새로 만드는 것

### 방법 B: reportlab + svglib 조합

reportlab 캔버스에 SVG 경로를 클리핑 마스크로 설정하고, 기존 디자인 PDF를 그 위에 배치한다.
- **장점**: Python만으로 구현 가능, svglib이 SVG 파싱 자동 처리
- **단점**: "기존 PDF를 reportlab 캔버스에 올리기"가 본질적으로 어려움. reportlab은 PDF 읽기용이 아님
- **비유**: 다른 공장(reportlab)에서 빵틀을 만드는데, 기존 빵(PDF)을 그 공장으로 옮기는 게 어려움

### 방법 C: 화이트 마스크 (반전 덮기)

패턴 바깥 영역을 흰색으로 덮어서 패턴 안쪽만 보이게 한다. 진정한 클리핑이 아닌 "덮기" 방식.
- **장점**: 구현 가장 간단, 현재 show_pdf_page 위에 바로 추가 가능
- **단점**: 패턴 바깥에 흰색이 남음(투명 X), 인쇄 시 흰색 잉크가 찍힐 수 있음
- **비유**: 쿠키 커터 대신 "틀 바깥을 흰 종이로 덮는 것" — 결과는 비슷해 보이지만 본질이 다름

### 방법 D: Illustrator ExtendScript

Illustrator의 네이티브 클리핑 마스크 기능을 스크립트로 자동화한다.
- **장점**: 가장 정확하고 자연스러운 결과, 디자이너가 수동으로 하는 것과 동일
- **단점**: Illustrator 설치 필수, 처리 속도 느림(파일당 5~10초), 초기 결정에서 Illustrator 의존도를 낮추기로 했음
- **비유**: 전문 제빵사(Illustrator)에게 직접 맡기는 것 — 결과는 최고지만 비용이 큼

### 방법 E: PDF W 연산자 직접 삽입

PDF 명세의 `W` (clip) 연산자를 콘텐츠 스트림에 직접 삽입한다. SVG 경로 좌표를 PDF 경로 명령으로 변환한다.
- **장점**: 가장 깔끔한 PDF 출력, CMYK 완벽 보존, 추가 라이브러리 불필요, CTM 직접 삽입(현재 검증 완료) 경험 활용
- **단점**: SVG -> PDF 좌표 변환 필요, PDF 좌표계(하단 원점) 주의
- **비유**: 빵틀 설계도(SVG)를 PDF 언어로 직접 번역해서 빵(디자인) 위에 얹는 것

## 5. 좌표 변환 문제 (A, E 방법의 핵심 난관)

SVG와 PDF는 좌표계가 다르다:

| | SVG | PDF |
|---|---|---|
| 원점 위치 | 좌상단 (왼쪽 위) | 좌하단 (왼쪽 아래) |
| Y축 방향 | 아래로 증가 | 위로 증가 |
| 단위 | px (기본), mm, pt 등 | pt (1/72 inch) |

**변환 공식**: `pdf_y = page_height - svg_y`

SVG 경로 명령(M, L, C, Z)은 PDF 경로 명령(m, l, c, h)과 **구조가 거의 동일**하다:
- SVG `M 10 20 L 30 40 C 50 60 70 80 90 100 Z`
- PDF `10 980 m 30 960 l 50 940 70 920 90 900 c h` (Y 반전, 페이지높이=1000 가정)

Bezier 곡선(C 명령)도 제어점의 Y만 반전하면 되므로, 곡선 형태는 정확히 보존된다. svgpathtools 라이브러리로 SVG 경로를 파싱하면 각 세그먼트(Line, CubicBezier 등)를 좌표 단위로 추출할 수 있다.

## 6. 다중 패턴 조각 처리

실제 의류 패턴은 앞판, 뒷판, 소매 등 **여러 조각**으로 구성된다. 두 가지 접근이 가능하다:

1. **조각별 개별 클리핑**: 각 조각마다 별도 클리핑 마스크 적용 -> 정밀하지만 복잡
2. **전체 윤곽선 한 번 클리핑**: SVG 전체의 외곽 경로로 한 번만 클리핑 -> 단순하지만 조각 사이 빈 공간에 디자인이 남을 수 있음

MVP에서는 **방법 2 (전체 윤곽선)**가 현실적이다. 조각 사이 빈 공간은 인쇄 시 잘려나가므로 실질적 문제가 되지 않는다.

## 7. 리스크 분석

| 리스크 | 심각도 | 대응 방안 |
|--------|--------|----------|
| SVG Bezier -> PDF 변환 부정확 | 중 | svgpathtools로 정밀 파싱, 단위 테스트로 검증 |
| 복잡한 곡선(오목/볼록)에서 클리핑 오류 | 중 | PDF even-odd fill rule 적용으로 해결 가능 |
| SVG viewBox/단위가 패턴마다 다름 | 높 | SVG 파싱 시 viewBox 정규화 로직 필수 |
| 클리핑 추가로 처리 시간 증가 | 낮 | 경로 변환은 수학 연산뿐, 1초 미만 예상 |
| 디자인과 패턴의 정렬(alignment) 문제 | 높 | 기준점(중심 또는 좌상단) 통일 규칙 필요 |

**가장 큰 리스크**: 디자인 PDF 위에 패턴 SVG를 **어디에 정렬할 것인가**. 두 파일의 아트보드 크기, 원점, 배치가 다를 수 있다. 이 정렬 규칙은 사용자와 함께 정의해야 한다.

## 8. 단계적 접근 (MVP -> 확장)

### 즉시 (3~5일): 방법 C - 화이트 마스크
- 현재 코드 위에 바로 추가 가능
- 패턴 바깥을 흰색으로 덮어 "시각적 클리핑" 효과
- 승화전사 인쇄에서 흰색 = 원단 색이므로 **실용적으로 문제없을 가능성 높음**
- 빠르게 결과를 보고 사용자 피드백 수집

### 단기 (1~2주): 방법 A 또는 E - 진정한 클리핑
- SVG 경로 -> PDF 클리핑 패스 변환 구현
- CMYK 완벽 보존 + 투명 클리핑
- CTM 직접 삽입 경험(이미 검증 완료)을 활용하면 난이도 절감
- **A와 E의 차이**: A는 PyMuPDF API(`page.draw_path`)를 쓰고, E는 콘텐츠 스트림에 직접 `W` 연산자를 삽입. 현재 CTM 직접 삽입 경험상 **E가 더 자연스러운 확장**

### 장기 (필요 시): 방법 D - Illustrator 연동
- 특수한 디자인(그라디언트 메시, 패턴 필 등)에서 최고 품질 필요 시
- 현재 아키텍처에서 Illustrator 연동은 "선택적 확장"으로 이미 설계됨

## 9. 추천 방향

**방법 E (PDF W 연산자 직접 삽입)를 최우선 추천한다.**

이유:
1. **현재 검증된 기술의 자연스러운 확장** -- CTM 직접 삽입(콘텐츠 스트림 조작)을 이미 프로토타입 검증했고, 같은 방식으로 `W` 클리핑 연산자를 추가하면 된다
2. **CMYK 완벽 보존** -- 원본 콘텐츠 스트림을 건드리지 않고 클리핑만 추가
3. **추가 의존성 최소** -- svgpathtools 하나만 추가 (SVG 경로 파싱용)
4. **Illustrator 불필요** -- 초기 결정(Illustrator 의존도 낮추기)과 일치
5. **처리 속도** -- 수학 연산만으로 좌표 변환, 파일당 1초 미만 예상

단, 시간이 부족하면 **방법 C (화이트 마스크)로 빠르게 MVP**를 만들고, 이후 E로 교체하는 것도 합리적이다. 승화전사 인쇄에서 흰색 = 원단 색이므로, 화이트 마스크도 실용적 결과를 낼 수 있다.

### 구현 순서 요약

```
[현재] 단순 비례 스케일링 (show_pdf_page)
  |  +3~5일
[Phase 1] 화이트 마스크로 빠른 검증 (선택적)
  |  +1~2주
[Phase 2] PDF W 연산자 클리핑 (본격 구현)
  |  필요 시
[Phase 3] Illustrator 연동 (특수 케이스)
```

```

100 500 m          % 시작점 (Move to)
200 600 l          % 직선 (Line to)
250 650 300 700 350 680 c  % Bezier 곡선 (Curve to)
100 500 l          % 시작점으로 복귀
W n                % 클리핑 적용 (W=clip, n=경로 지우기)

% 이후 기존 디자인 콘텐츠가 이 클리핑 영역 안에서만 렌더링됨
q 0.85 0 0 0.90 10 20 cm  % 기존 CTM 스케일링
... (원본 디자인 콘텐츠) ...
Q
```
