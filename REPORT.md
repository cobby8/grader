## 1. 종합 결론

### 판정: 조건부 가능

이 프로그램은 **기술적으로 구현 가능**합니다. 다만 한 가지 중요한 조건이 있습니다:

> **AI 파일(Adobe Illustrator 원본)을 프로그램이 직접 읽고 조작하는 것은 매우 어렵고 불안정합니다.**  
> 대신, **Adobe Illustrator를 자동 조종(스크립팅)**하는 방식이 현실적인 해법입니다.

쉬운 비유로 설명하면:
- **직접 파싱 방식** = 외국어 책을 사전 없이 읽으려는 것 (AI 파일 포맷은 비공개이고 복잡)
- **Illustrator 스크립팅 방식** = 통역사(Illustrator)에게 "이 부분을 이렇게 바꿔줘"라고 지시하는 것

나머지 요구사항(SVG 파싱, CMYK 색상 보존, PDF/EPS 출력, 사이즈별 스케일링)은 모두 검증된 기술로 구현 가능합니다.

## 2. 확정 워크플로우 (5단계)

사용자와 협의하여 확정된 프로그램의 전체 작업 흐름입니다. 이 5단계가 프로그램이 처리하는 핵심 파이프라인입니다.

> **비유**: 이 5단계는 "공장의 생산 라인"과 같습니다. 재료(디자인+패턴)가 들어가면 완성품(인쇄용 파일)이 나옵니다.

### 전체 흐름도

```
[1단계]              [2단계]              [3단계]           [4단계]              [5단계]
프리셋 패턴 등록  ->  기준사이즈          ->  주문서에서      ->  사이즈별          ->  전 과정
(SVG 파일)          디자인 파일 등록         필요 사이즈         그레이딩 파일         CMYK 색상
                    (PDF 추천)              인식                생성 (PDF 출력)       유지
```

### 각 단계 상세

| 단계 | 이름 | 입력 | 처리 내용 | 출력 | 비유 |
|------|------|------|----------|------|------|
| **1단계** | 프리셋 패턴 등록 | SVG 파일 | 의류의 각 조각(앞판, 뒷판, 소매 등) 윤곽선을 등록. 사이즈별 치수 정보 포함 | 패턴 프리셋 데이터 | 옷본(패턴지)을 보관함에 정리하는 것 |
| **2단계** | 기준사이즈 디자인 등록 | PDF 파일 (추천) | 디자이너가 만든 기준 사이즈(예: L) 디자인을 등록. 프로그램이 이 파일을 읽어서 각 패턴 조각에 맞게 처리할 수 있어야 함 | 등록된 기준 디자인 | 원본 벽지(디자인)를 가져오는 것 |
| **3단계** | 주문서 사이즈 인식 | 주문서 (엑셀/수동) | 주문서에서 필요한 사이즈 목록을 추출. MVP에서는 수동 선택, 이후 엑셀 자동 인식 | 필요 사이즈 목록 | 주문표에서 "S 5장, M 10장" 읽는 것 |
| **4단계** | 그레이딩 파일 생성 | 패턴 + 디자인 + 사이즈 | 기준 사이즈를 각 목표 사이즈로 스케일링하여 인쇄용 파일 생성 | 사이즈별 PDF 파일 | 원본 벽지를 각 방 크기에 맞게 오려 붙이는 것 |
| **5단계** | CMYK 색상 유지 | (전 과정) | 1~4단계 모든 과정에서 인쇄용 CMYK 색상이 변하지 않도록 보장 | CMYK 검증 완료 | 요리 레시피 비율이 접시 크기와 무관하게 유지되는 것 |

### 1단계와 2단계의 차이 (중요!)

- **1단계 (패턴 = SVG)**: 옷의 "모양"만 필요. 앞판이 어떤 형태인지, 소매가 어떤 윤곽인지. 색상 불필요, 윤곽선만 있으면 됨.
- **2단계 (디자인 = PDF)**: 옷에 입힐 "그림"이 필요. 풀프린트 무늬, 로고 위치, 색상 정보 등. CMYK 색상이 핵심.

> **비유**: 1단계는 "쿠키 틀(모양)"을 등록하는 것이고, 2단계는 "아이싱(장식 디자인)"을 등록하는 것입니다.

## 3. 기준 디자인 파일 형식 분석

### 3.1 핵심 질문

> 디자이너가 만든 기준사이즈 디자인 파일을 어떤 형식으로 올리는 것이 최적인가?

이 파일은 단순히 보관하는 것이 아니라, **프로그램이 읽어서 각 패턴 조각에 맞게 자르고(클리핑), 크기를 조절(스케일링)해야** 합니다. 그래서 "프로그래밍으로 조작 가능한가"가 매우 중요한 기준입니다.

### 3.2 후보 형식 비교표

| 비교 기준 | AI 파일 (Illustrator 원본) | EPS 파일 | PDF 파일 |
|----------|--------------------------|----------|---------|
| **CMYK 보존 정확도** | 완벽 (원본이므로) | 완벽 (CMYK 네이티브) | 완벽 (CMYK 네이티브) |
| **프로그래밍 조작** | Illustrator ExtendScript 필요 (간접 조작) | Python으로 가능하나 라이브러리 제한적 | Python으로 가능 (라이브러리 풍부) |
| **벡터 보존** | 완벽 | 완벽 | 완벽 |
| **디자이너 편의성** | 가장 편함 (원본 저장) | "다른 이름으로 저장"에서 선택 | "다른 이름으로 저장"에서 선택 |
| **파일 용량** | 가장 큼 (편집 데이터 포함) | 중간 | 가장 작음 |
| **공장/인쇄기 호환** | Illustrator 필요 | 거의 모든 인쇄 장비 | 거의 모든 장비 |
| **레이어/오브젝트 구조** | 완벽 보존 | 레이어 정보 손실 가능 | Illustrator에서 "레이어 보존" 옵션으로 저장 시 보존 |
| **Python 처리 라이브러리** | 없음 (직접 파싱 불가) | ghostscript, pstoedit 등 제한적 | PyMuPDF, pikepdf, reportlab, pdfplumber 등 풍부 |

### 3.3 각 형식의 장단점 상세

#### AI 파일 (Adobe Illustrator 원본)

- **장점**: 모든 정보(레이어, 효과, 색상, 오브젝트 구조)가 완벽히 보존됨
- **단점**: Adobe 비공개 형식이라 프로그램이 직접 읽을 수 없음. 반드시 Illustrator를 거쳐야 조작 가능
- **조작 방식**: Illustrator ExtendScript로 간접 조작 (Illustrator가 반드시 실행되어야 함)
- **비유**: "원본 요리 노트"이지만, 특정 셰프(Illustrator)만 읽을 수 있는 필체로 쓰여 있음

#### EPS 파일 (Encapsulated PostScript)

- **장점**: 오래된 인쇄 업계 표준. CMYK 완벽 지원. 대부분의 인쇄 장비와 호환
- **단점**: PostScript 기반이라 프로그래밍으로 내부 오브젝트를 개별 조작하기가 어려움. Python에서 EPS를 파싱해서 개별 요소에 접근하는 라이브러리가 부족함. Ghostscript으로 변환은 가능하지만 "이 레이어의 이 오브젝트만 스케일링"같은 세밀한 조작은 제한적
- **비유**: "인쇄소용 완성 원고"이지만, 원고 안의 개별 글자를 바꾸기가 어려움

#### PDF 파일 (추천)

- **장점**:
  - CMYK 완벽 지원 (PDF 자체가 인쇄 표준)
  - Python 라이브러리가 가장 풍부 (PyMuPDF/fitz, pikepdf, pdfplumber, reportlab)
  - PyMuPDF로 PDF 내부 벡터 경로/오브젝트에 접근하여 추출, 변환, 재구성 가능
  - Illustrator에서 PDF 저장 시 "Illustrator 편집 기능 보존" 옵션을 켜면 레이어/오브젝트 구조 유지
  - 파일 용량이 가장 작음
  - 모든 장비, 모든 OS에서 열 수 있음
- **단점**: 
  - Illustrator 편집 기능 보존 옵션 없이 저장하면 레이어 구조가 평탄화(flatten)될 수 있음
  - 일부 특수 효과가 래스터화될 수 있음 (하지만 승화전사 디자인에서는 드문 케이스)
- **비유**: "만국 공용어로 쓴 레시피". 누구나 읽을 수 있고, 재료(색상)도 정확하며, 프로그램도 쉽게 처리 가능

### 3.4 Python에서 PDF 조작이 가능한 구체적 방법

PDF를 기준 디자인 형식으로 선택하면, Python으로 다음 작업이 가능합니다:

| 작업 | 사용 라이브러리 | 설명 |
|------|--------------|------|
| PDF 열기 및 페이지 읽기 | PyMuPDF (fitz) | PDF를 열어서 각 페이지의 내용에 접근 |
| 벡터 경로 추출 | PyMuPDF | 페이지 내 벡터 그래픽의 경로(path) 데이터를 추출 |
| 클리핑 (영역 자르기) | PyMuPDF, reportlab | 패턴 조각 윤곽선에 맞게 디자인을 잘라냄 |
| 스케일링 (크기 변환) | PyMuPDF, reportlab | 좌표에 비율을 곱해서 사이즈 변환 |
| CMYK 색상 읽기 | pikepdf, PyMuPDF | 색상 공간(CMYK/RGB) 확인 및 색상값 추출 |
| 새 PDF 생성 | reportlab | 변환된 결과를 새 PDF로 출력 (CMYK 유지) |

> **중요**: PyMuPDF는 PDF 내부의 벡터 그래픽을 "수학적 좌표 데이터"로 추출할 수 있습니다. 이 데이터를 가지고 스케일링/클리핑 연산을 수행한 뒤, ReportLab으로 새 PDF를 생성하면 됩니다.

### 3.5 추천: PDF 파일 (Illustrator 호환 PDF)

**PDF를 기준 디자인 파일 형식으로 추천합니다.** 이유:

1. **프로그래밍 조작이 가장 용이**: Python 라이브러리(PyMuPDF, reportlab 등)로 직접 열어서 벡터 데이터 추출, 클리핑, 스케일링이 모두 가능
2. **CMYK 완벽 보존**: PDF는 인쇄 표준이므로 CMYK 색상 공간을 네이티브로 지원
3. **Illustrator 없이도 처리 가능**: AI 파일은 Illustrator가 반드시 필요하지만, PDF는 Python만으로 처리 가능 (Illustrator 의존도 감소)
4. **파일 용량이 가장 작음**: 네트워크 전송이나 저장 공간에 유리
5. **범용 호환성**: 공장 인쇄기, 모든 OS, 모든 뷰어에서 열 수 있음

**저장 시 디자이너가 지켜야 할 설정**:
- Illustrator에서 "다른 이름으로 저장" -> PDF 선택
- **"Illustrator 편집 기능 보존" 체크** (레이어/오브젝트 구조 유지)
- 색상 모드: CMYK 유지 (RGB로 변환하지 않기)
- 호환성: Acrobat 5 이상

### 3.6 AI 파일의 역할 변경

기존 계획에서는 AI 파일을 직접 Illustrator 스크립팅으로 조작하는 것이 핵심이었습니다. 새 워크플로우에서는:

- **AI 파일**: 디자이너의 원본 작업 파일로만 사용 (보관 용도)
- **PDF 파일**: 프로그램에 등록하는 기준 디자인 파일 (처리 대상)
- **Illustrator 스크립팅**: 여전히 보조적으로 활용 가능 (복잡한 효과가 있는 디자인의 경우 Illustrator를 통해 PDF로 변환하는 전처리 단계에서 사용)

> 이 변경으로 **Illustrator 필수 설치 조건이 완화**됩니다. 기본적으로 Python만으로 처리 가능하고, Illustrator는 선택적 기능이 됩니다.

## 4. 주문서 사이즈 인식 방식 분석

### 4.1 3단계의 역할

워크플로우 3단계에서는 "어떤 사이즈를 몇 장 만들어야 하는지"를 파악해야 합니다. 이 정보가 있어야 4단계에서 해당 사이즈만 생성할 수 있습니다.

### 4.2 주문서 입력 방식 후보

| 방식 | 설명 | 장점 | 단점 | MVP 적합도 |
|------|------|------|------|-----------|
| **수동 선택 (추천 - MVP)** | UI에서 체크박스로 필요 사이즈 선택 | 가장 단순, 즉시 구현 가능 | 매번 수동 입력 필요 | 최적 |
| **엑셀 파일 업로드** | .xlsx 파일을 읽어서 사이즈/수량 자동 추출 | 대량 주문 처리에 편리 | 엑셀 형식이 업체마다 다를 수 있음 | 2차 개발 |
| **CSV 텍스트 입력** | 간단한 텍스트 형식으로 사이즈 목록 입력 | 구현 간단, 형식 통일 용이 | 디자이너에게 익숙하지 않을 수 있음 | 2차 개발 |
| **이미지(OCR) 인식** | 주문서 사진을 찍어 텍스트 인식 | 기존 서류 그대로 활용 | 인식 정확도 문제, 구현 복잡 | 불필요 |

### 4.3 수량 정보는 필요한가?

- **파일 생성 관점**: 수량은 불필요. S사이즈가 5장이든 100장이든, 인쇄용 PDF 파일은 1개만 만들면 됨 (인쇄소에서 수량만큼 반복 인쇄)
- **관리 관점**: 수량 정보가 있으면 "S 5장, M 10장, L 15장" 같은 주문 요약을 만들 수 있어 편리
- **결론**: MVP에서는 **사이즈 선택만** 구현. 수량 관리는 이후 확장

### 4.4 추천: 2단계 접근

**1차 (MVP)**: UI에서 사이즈 체크박스 선택
```
[v] S    [ ] M    [v] L (기준)    [v] XL    [ ] 2XL
[전체 선택] [기준 사이즈 제외]
```

**2차 (확장)**: 엑셀 파일 업로드 자동 인식
- Python의 openpyxl 라이브러리로 .xlsx 파일 읽기
- 사이즈 컬럼을 자동 탐지하거나, 사용자가 컬럼을 지정
- 인식된 사이즈 목록을 UI에 표시하여 확인 후 진행

## 5. 기술 타당성 분석

### 5.1 AI 파일을 프로그래밍으로 읽고 조작할 수 있는가?

| 방법 | 가능 여부 | 설명 |
|------|----------|------|
| AI 파일 직접 파싱 | 매우 어려움 | AI 파일은 Adobe의 비공개 형식. 내부에 PDF와 PGF(Adobe 고유 형식) 데이터가 섞여 있음. 오픈소스 파서(illustrator-parser-pdfcpu)가 존재하지만, 완전한 편집은 불가 |
| Aspose 라이브러리 | 부분 가능 | AI 파일을 읽어서 PNG/PSD 등으로 변환은 가능하지만, CMYK 벡터 데이터를 온전히 조작하기는 제한적 |
| **Illustrator 스크립팅 (추천)** | **가능** | Illustrator의 ExtendScript/JavaScript API를 통해 파일 열기, 요소 조작, 스케일링, 내보내기를 모두 자동화 가능. **CMYK 색상값도 완벽 보존** |

> **비유**: AI 파일을 직접 읽는 것은 "잠긴 금고를 열쇠 없이 여는 것"이고, Illustrator 스크립팅은 "금고 주인(Illustrator)에게 열어달라고 부탁하는 것"입니다.

**결론**: Illustrator 스크립팅 방식을 사용하면 AI 파일 조작은 **가능**합니다. 단, 사용자 PC에 Adobe Illustrator가 설치되어 있어야 합니다.

### 5.2 CMYK 색상값을 손실 없이 유지하면서 스케일링할 수 있는가?

**가능합니다.**

- **벡터 그래픽의 특성**: 벡터는 수학 공식으로 그림을 표현하므로, 크기를 바꿔도 품질 손실이 없습니다. 사진(래스터)과 달리 확대해도 깨지지 않습니다.
- **CMYK 보존**: Illustrator 스크립팅으로 스케일링하면 색상 정보가 그대로 유지됩니다. 색상 모드 변환(RGB <-> CMYK)을 하지 않으면 색상값은 절대 바뀌지 않습니다.
- **PDF 출력 시**: ReportLab(Python 라이브러리)은 CMYK 색상 모델을 네이티브로 지원하며, CMYKColor 클래스를 통해 정밀한 색상 제어가 가능합니다.

> **비유**: CMYK 색상값은 "요리 레시피의 재료 비율" 같은 것입니다. 접시 크기(사이즈)를 바꿔도 레시피 비율(색상값)은 변하지 않습니다.

### 5.3 SVG 파싱 및 좌표 변환

**가능합니다.**

SVG는 공개 표준이라 파싱이 매우 쉽습니다. Python에만 5개 이상의 검증된 라이브러리가 있습니다:

| 라이브러리 | 용도 |
|-----------|------|
| **svgelements** | SVG를 기하학적 데이터로 변환 (좌표, 경로 추출) |
| **svgpathtools** | SVG 경로 조작 및 베지에 곡선 분석 |
| **svglib** | SVG를 ReportLab 그래픽으로 변환 (PDF 출력에 유리) |

> **비유**: SVG 파싱은 "엑셀 파일 읽기"처럼 표준화된 작업입니다. 어려울 것이 없습니다.

### 5.4 벡터 그래픽 스케일링의 정확도

**매우 높습니다.**

벡터 그래픽은 점과 점 사이의 수학적 관계로 정의되므로, 스케일링은 단순히 좌표값에 비율을 곱하는 것입니다. 이론적으로 무한히 정확하며, 실무적으로도 소수점 이하 충분한 정밀도를 유지합니다.

### 5.5 PDF/EPS 출력 시 CMYK 보존 방법

| 도구 | CMYK 지원 | 설명 |
|------|----------|------|
| **ReportLab (Python)** | 네이티브 지원 | CMYKColor, PCMYKColor 클래스 제공. PDF/EPS 모두 출력 가능 |
| **Ghostscript** | 완벽 지원 | PDF/EPS 간 변환, ICC 프로파일 적용, CMYK 색상 공간 강제 가능 |
| **Illustrator 스크립팅** | 완벽 지원 | Illustrator가 직접 PDF/EPS로 내보내므로 가장 안전 |

## 7. 핵심 기술적 도전과 해결 방안

### 7.1 AI 파일 포맷 파싱의 어려움

**문제**: AI 파일은 Adobe의 비공개(독점) 형식입니다. 내부에 PDF 호환 데이터와 Adobe 고유 PGF 데이터가 섞여 있어서, 외부 프로그램이 완벽하게 읽기 어렵습니다.

**해결 방안: Illustrator를 "중개인"으로 활용**

```
[우리 프로그램] --명령--> [Adobe Illustrator] --처리--> [결과 파일]
```

1. 프로그램이 Illustrator에게 "이 AI 파일을 열어줘"라고 명령
2. Illustrator가 파일을 열면, 스크립트를 통해 내부 요소에 접근
3. 필요한 조작(스케일링, 클리핑 등)을 수행
4. Illustrator가 PDF/EPS로 내보내기

이 방식의 핵심 장점: **AI 파일의 모든 정보(레이어, 색상, 효과 등)가 100% 보존**됩니다.

### 7.2 CMYK 색상 공간 처리

**문제**: 컴퓨터 화면은 RGB(빛의 삼원색), 인쇄는 CMYK(잉크의 사원색)를 사용합니다. 변환 과정에서 색상이 달라질 수 있습니다.

> **비유**: RGB는 "빛으로 색을 만드는 것"(TV 화면), CMYK는 "물감으로 색을 만드는 것"(프린터). 같은 빨간색이라도 빛의 빨강과 물감의 빨강은 다릅니다.

**해결 방안**:
- 처음부터 끝까지 CMYK 색상 공간을 유지하고, RGB로 변환하지 않음
- Illustrator 스크립팅에서 색상 모드를 CMYK로 고정
- PDF 출력 시 ReportLab의 CMYKColor 클래스 사용
- Ghostscript으로 최종 검증 (ICC 프로파일 적용)

### 7.3 패턴 조각에 맞춘 그래픽 클리핑/마스킹

**문제**: 하나의 디자인(예: 풀프린트 무늬)을 옷의 각 조각(앞판, 뒷판, 소매) 모양에 맞게 잘라내야 합니다.

> **비유**: 큰 벽지(디자인)를 방의 벽 모양(패턴 조각)에 맞게 오려 붙이는 것과 같습니다.

**해결 방안**:
- 패턴 조각의 윤곽선(외곽선)을 "클리핑 마스크"로 사용
- 클리핑 마스크 = "이 영역 안의 것만 보여줘"라는 명령
- Illustrator 스크립팅에서 클리핑 마스크 자동 적용 가능
- SVG 좌표로 패턴 조각의 위치와 크기를 정밀 제어

### 7.4 사이즈별 스케일링 알고리즘

**문제**: 옷 사이즈를 키울 때, 모든 부분이 같은 비율로 커지는 것이 아닙니다. 예를 들어 S에서 XL로 갈 때 가슴 너비는 많이 커지지만 칼라 높이는 조금만 커집니다.

> **비유**: 사람이 성장할 때 머리는 조금만 커지고 몸통은 많이 커지는 것처럼, 옷 패턴도 부위마다 다른 비율로 스케일링해야 합니다.

**해결 방안: 부위별 차등 스케일링**

```
패턴 프리셋 데이터 예시:
+---------------------------------------------------+
| 의류: 반팔 티셔츠                                    |
|                                                     |
| 사이즈  | 앞판(가로x세로) | 뒷판      | 소매        |
| S       | 45x65 cm      | 46x65 cm | 20x28 cm   |
| M       | 48x67 cm      | 49x67 cm | 21x29 cm   |
| L (기준)| 51x69 cm      | 52x69 cm | 22x30 cm   |
| XL      | 54x71 cm      | 55x71 cm | 23x31 cm   |
| 2XL     | 57x73 cm      | 58x73 cm | 24x32 cm   |
+---------------------------------------------------+
```

- 각 패턴 조각별로 독립적인 스케일 비율 계산
- 기준 사이즈(L) 대비 목표 사이즈의 비율을 패턴 조각마다 개별 적용
- 디자인 요소(로고, 번호 등)의 위치도 비율에 따라 재배치

### 7.5 대용량 벡터 파일 처리 성능

**문제**: 복잡한 디자인은 수천 개의 벡터 경로를 포함할 수 있으며, 13개 사이즈를 모두 생성하면 처리 시간이 길어질 수 있습니다.

**해결 방안**:
- **병렬 처리**: 각 사이즈를 독립적으로 동시에 처리 (Python multiprocessing)
- **캐싱**: 한 번 파싱한 SVG 데이터를 메모리에 보관하여 재사용
- **진행 표시**: UI에 진행률 바를 표시하여 사용자가 기다리기 편하게
- **예상 처리 시간**: 일반적인 디자인 기준, 13개 사이즈 전체 생성에 1~3분 예상

| **출력 모듈** | PDF/EPS 파일 생성, CMYK 보존, 용량 최적화 | 인쇄소에 보낼 최종 파일 만들기 |
| **인증/권한 모듈** | 로그인, 계정 생성, 권한 부여 | 건물 출입 카드 시스템 |

### 8.3 데이터 흐름 (확정 워크플로우 기반)

```
[1단계] 관리자가 패턴 프리셋을 SVG 파일로 등록
        (의류 종류별 각 조각의 윤곽선 + 사이즈별 치수)
    |
    v
[2단계] 디자이너가 기준사이즈 디자인을 PDF 파일로 등록
        (Illustrator에서 CMYK PDF로 저장한 디자인)
    |
    v
[3단계] 주문서에서 필요한 사이즈를 인식
        (MVP: UI에서 수동 체크박스 선택)
        (확장: 엑셀 파일 자동 인식)
    |
    v
[4단계] Python 엔진이 자동 처리:
        - PDF에서 벡터 데이터 추출 (PyMuPDF)
        - 패턴 조각별 클리핑 마스크 적용
        - 사이즈별 스케일링 연산
        - CMYK 유지한 채 PDF 출력 (ReportLab)
    |
    v
[5단계] 사이즈별 인쇄용 파일 생성 완료! (CMYK 검증 완료)
        예: "반팔티_앞판_S.pdf", "반팔티_앞판_M.pdf" ... "반팔티_앞판_5XL.pdf"
```

## 9. 개발 로드맵

### 9.1 MVP(최소 기능 제품) 범위

MVP는 "가장 핵심적인 기능만 먼저 만들어서 쓸 수 있는 버전"입니다.

**MVP에 포함** (확정 워크플로우 1~5단계 기본 구현):
- [1단계] 패턴 프리셋 등록 및 관리 (SVG, 1개 의류 종류)
- [2단계] 기준사이즈 디자인 PDF 파일 등록 및 파싱
- [3단계] UI에서 필요 사이즈 수동 선택 (체크박스)
- [4단계] Python 엔진으로 사이즈별 그레이딩 파일 자동 생성
- [5단계] 전 과정 CMYK 색상 유지
- 기본 UI (Tauri + React)

**MVP에서 제외 (이후 추가)**:
- 계정/권한 관리 시스템
- 엑셀 주문서 자동 인식 (3단계 확장)
- EPS 출력
- 다양한 의류 종류 패턴
- 용량 최적화
- 배치 처리 (여러 디자인 한번에)

### 9.2 단계별 개발 계획

| 개발 단계 | 내용 (워크플로우 단계) | 예상 기간 | 우선순위 |
|------|------|----------|---------|
| **1단계: 기반 구축** | Tauri 프로젝트 세팅, UI 기본 틀, 프로젝트 구조 | 1~2주 | 필수 |
| **2단계: 패턴 시스템** | [WF 1단계] 패턴 프리셋 데이터 구조 설계, SVG 파싱, 등록/조회 UI, 로컬 저장 | 1~2주 | 필수 |
| **3단계: 디자인 파일 처리** | [WF 2단계] PDF 파일 등록, PyMuPDF로 벡터 데이터 추출, CMYK 색상 공간 검증 | 2~3주 | 필수 |
| **4단계: 사이즈 선택 + 스케일링** | [WF 3,4단계] 사이즈 선택 UI, 스케일링 연산, 클리핑 마스크 적용, PDF 출력 | 2~3주 | 필수 |
| **5단계: CMYK 보존 + 출력** | [WF 5단계] 전 과정 CMYK 검증, ReportLab PDF 출력, Ghostscript 검증 | 1~2주 | 필수 |
| **6단계: 통합 테스트** | 전체 워크플로우 테스트, 실제 디자인으로 검증, 버그 수정 | 2주 | 필수 |
| **--- MVP 완료 선 ---** | | **총 9~14주** | |
| **7단계: 계정/권한** | Supabase 연동, 로그인, 역할별 권한, 마스터 페이지 | 2~3주 | 높음 |
| **8단계: 주문서 자동 인식** | [WF 3단계 확장] 엑셀 파일 업로드, 사이즈/수량 자동 추출 (openpyxl) | 1~2주 | 높음 |
| **9단계: 확장** | 다양한 의류 패턴 추가, EPS 출력, 배치 처리, Illustrator 연동 (선택) | 2~3주 | 보통 |
| **10단계: 최적화** | 성능 개선, UI 다듬기, 사용자 피드백 반영 | 2주 | 보통 |

**총 예상 기간**: MVP 약 3~4개월, 전체 완성 약 5~7개월

> **참고**: 기존 계획에서는 Illustrator 연동이 4단계(필수)였으나, PDF 기반 처리로 변경되면서 Illustrator 연동은 9단계(선택 확장)으로 이동했습니다. 이로 인해 Illustrator 필수 설치 조건이 완화되었습니다.

### 9.3 우선순위 정리

```
[즉시 필요]  패턴 프리셋(SVG) > 디자인 파일(PDF) 처리 > 사이즈 선택 + 스케일링 > CMYK PDF 출력
[이후 추가]  계정/권한 관리 > 엑셀 주문서 자동 인식 > 다양한 의류 패턴 > EPS 출력
[나중에]     Illustrator 연동(선택) > 배치 처리 > 성능 최적화 > UI 고도화
```

## 10. 리스크 분석

### 10.1 기술적 리스크

| 리스크 | 심각도 | 발생 가능성 | 대응 방안 |
|--------|-------|------------|----------|
| PyMuPDF의 PDF 벡터 추출 정확도 | 높음 | 낮음 | 복잡한 디자인에서 벡터 경로 손실 가능. 사전 테스트로 검증, 문제 시 Illustrator 연동 대안 |
| PDF에서 레이어 구조 손실 | 중간 | 중간 | "Illustrator 편집 기능 보존" 옵션으로 저장하도록 안내. 미보존 시 평탄화된 데이터로 처리 |
| 대용량 PDF 파일 처리 지연 | 중간 | 중간 | 진행률 표시, 백그라운드 처리, 타임아웃 설정 |
| Python 번들링 복잡성 | 중간 | 중간 | PyInstaller 또는 embedded Python으로 앱에 내장. 또는 Python을 사전 설치 요구 |
| Illustrator 미설치 환경 | 낮음 (변경) | 중간 | PDF 기반 처리로 변경되어 Illustrator 필수 아님. 선택적 확장 기능에서만 필요 |

### 10.2 PDF 파일 호환성 리스크 (변경: 기존 AI 파일 -> PDF 파일 기준)

| 상황 | 리스크 | 대응 |
|------|--------|------|
| PDF에 래스터 이미지(사진) 포함 | 중간 | 래스터 부분은 별도 처리, 해상도 유지 경고 표시. 스케일링 시 해상도 저하 주의 |
| PDF에 특수 효과(그라데이션, 블렌드) 포함 | 중간 | PyMuPDF로 처리 가능한 범위 확인 필요. 처리 불가 시 Illustrator 연동 대안 |
| "Illustrator 편집 기능 보존" 없이 저장된 PDF | 중간 | 레이어/오브젝트 구조 없이 평탄화됨. 단일 벡터로 처리하되, 정밀한 오브젝트별 조작은 제한될 수 있음 |
| 디자이너가 RGB 모드로 PDF 저장 | 높음 | PDF 등록 시 색상 공간 자동 검사. RGB 감지 시 경고 메시지 표시, CMYK로 재저장 안내 |

### 10.3 색상 정확도 리스크

| 리스크 | 대응 |
|--------|------|
| RGB -> CMYK 변환 시 색상 차이 | **변환하지 않음**. 처음부터 CMYK로 작업하고 끝까지 유지 |
| PDF 뷰어에서 색상이 다르게 보임 | 이것은 모니터(RGB) 표시의 한계. 실제 인쇄물과는 관계없음을 안내 |
| 인쇄기 종류에 따른 색상 차이 | ICC 프로파일 설정 기능 제공. 공장별 프로파일 적용 가능 |
| 스팟 컬러(특색) 처리 | Illustrator 스크립팅으로 스팟 컬러 정보도 보존 가능 |

### 10.4 비기술적 리스크

| 리스크 | 대응 |
|--------|------|
| 디자이너 학습 비용 | 직관적 UI + 사용 가이드 영상 제작 |
| 패턴 치수 데이터 입력 오류 | 입력값 검증, 시각적 미리보기 제공 |
| Adobe Illustrator 라이선스 비용 | 이미 업무용으로 사용 중이므로 추가 비용 없음 (확인 필요) |
