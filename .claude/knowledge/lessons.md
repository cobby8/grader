### [2026-04-21] SVG 변환 파이프라인 검증의 핵심: Idempotent 테스트 + 시각 확인 단계
- **분류**: lesson
- **발견자**: pm + developer (svg_normalizer 개발 + 원본 U넥 작업 회고)
- **내용**: U넥 양면유니폼 SVG 변환 작업에서 사용자 롤백 2번 발생(단순 Tx swap 실패 → 가운데 417 겹침, 작은 절단선이 패턴 따라 이동). 이 시행착오의 공통 원인은 "변환 후 시각 확인까지 가서야 문제 발견"이었음. **두 가지 안전장치 도입**: (1) **Idempotent 테스트**: 같은 입력을 변환기에 2회 통과시켜 결과 hash가 일치해야 통과. svg_normalizer 회귀 검증에서 비트단위 동일 결과 확인됨. (2) **변환 전 시뮬레이션 단계**: 실제 파일 수정 전 예상 좌표를 미리 출력해 사용자가 사전 검토. 이 두 가지로 변환 파이프라인의 신뢰성 비약적 향상. 교훈: **순수 함수형 변환 도구는 항상 idempotent여야 한다**, 그리고 **사용자 시각 확인이 필요한 작업은 변환 전 시뮬레이션이 필수다**.

### [2026-04-21] AI 파일은 첫 10바이트 헤더로 변환 도구를 분기해야 한다 (PDF vs PostScript)
- **분류**: lesson
- **발견자**: pm (외부 작업, G드라이브 63개 AI→SVG 변환)
- **내용**: Illustrator `.ai` 파일은 두 종류가 공존한다. (1) **PDF 호환 모드**로 저장된 AI: 헤더가 `%PDF-1.4`로 시작하며 PyMuPDF가 직접 열 수 있어 `page.get_svg_image(text_as_path=True)`로 SVG 추출 가능(11KB 수준 — 기존 SVG와 동일 포맷). (2) **PostScript 원본 모드**: 헤더 `%!PS-Adobe`로 시작하며 Adobe 전용 PostScript 확장(`--dict:X/Y(ro)(G)--` 등)을 포함해 **Ghostscript/Inkscape/PyMuPDF 모두 열 수 없음**. 실제 배치에서 63개 중 7개(11%)가 PostScript 원본, 대부분 마스터/템플릿 파일(날짜 찍힌 `(FG26-03-05)` 등). **해결**: Illustrator COM(`win32com`) 또는 기존 `run_illustrator_script` JSX로 먼저 PDF 호환 형식으로 재저장(`PDFSaveOptions.Compatibility=5` = PDF 1.4) 후 PyMuPDF로 변환. 교훈: **AI 배치 변환 파이프라인은 반드시 헤더 바이트 검사(`open(f,'rb').read(10)`) → 분기 처리**가 필요하다. 헤더 검사 없이 단일 도구로 처리하면 11% 실패가 예측 불가능하게 발생한다. grader에 AI→SVG 변환 기능 추가 시 이 분기 로직이 핵심.
- **참조횟수**: 0

### [2026-04-21] SVG path 분류는 x_min(위치)보다 width(크기)가 더 안정적인 기준이다
- **분류**: lesson
- **발견자**: developer (svg_normalizer 4그룹 구조 버그 수정)
- **내용**: U넥 양면유니폼 SVG는 두 가지 구조 공존 — (1) **원본 6 path** — 큰패턴 1 + 작은패턴 1 + 절단선 4 (변환 전), (2) **4그룹 12 path** — 큰패턴 위/아래 2 + 작은패턴 위/아래 2 + 절단선 8 (변환 후). 초기 `_extract_pattern_paths`는 `x_min 작으면 큰패턴`으로 분류했는데, 변환된 SVG는 큰패턴이 좌측(x=313)에 위치해 좌우 관계가 뒤집혀 분류 실패 → 좌측 큰패턴 2개 누락 → 재변환 시 좌표가 viewBox 밖. **수정**: (1) 패턴과 절단선을 높이로 분리(높이<5=절단선), (2) 4그룹이면 y_min 기준 위쪽 쌍만 채택, (3) 큰/작은 결정은 **폭(width) 우선 비교** — 폭이 더 큰 쪽이 큰패턴(앞판). 교훈: **"위치(x_min)"는 변환 전후로 뒤집힐 수 있지만 "크기(width)"는 사이즈만 다를 뿐 구조가 같으면 대소 관계 유지된다**. 멱등성 필수 변환에서 분류 기준은 **"변환 불변량"을 우선**해야 한다.
- **참조횟수**: 0

### [2026-04-16] ExtendScript 문서 간 이동은 clipboard 대신 duplicate가 기본
- **분류**: lesson
- **발견자**: developer (버그 B 수정)
- **내용**: Illustrator ExtendScript에서 source document의 아이템을 target document로 옮길 때 가장 직관적인 `copy/paste` 패턴은 Illustrator 앱 전역 상태(AICB clipboard)에 의존하기 때문에, 중간에 다른 문서를 열거나 닫으면 간헐적으로 clipboard가 무효화되는 위험이 있다. 특히 Illustrator는 **단일 인스턴스** 앱이라 이전 실행의 clipboard까지 섞이면서 재현도 불규칙하다. **대안**: `PageItem.duplicate(targetContainer, ElementPlacement.PLACEATEND)`는 clipboard를 거치지 않고 문서 간 직접 복제를 수행하며, 이미 grading.jsx STEP 7의 `path.duplicate(layerFill)`에서 검증된 방식이다. 제약: source document가 duplicate 호출 시점에 **열려 있어야** 원본 PageItem 참조가 유효. 교훈: 문서 간 이동 로직은 **처음부터 duplicate 패턴을 기본으로** 설계하고, clipboard는 "선택 복원이 필요한 경우에만" 제한적으로 사용한다.

### [2026-04-16] 회귀 누적 상태는 revert로 "기저점 확보 → 한 걸음씩 재도입"이 정답
- **분류**: lesson
- **발견자**: planner-architect (grading.jsx 4커밋 누적 회귀 감사)
- **내용**: grading.jsx에 c52d80f~06b16fa 4개 커밋이 누적되며 **3개 독립 버그**(STEP 8B 문서좌표 오비교, paste=0 격번, D1 아트보드 초과)가 동시 존재. 단일 hotfix(3개 동시 수정) 대신 **4커밋 전체 revert → 베이스라인 실측 → 기능 1개씩 재도입** 방식을 권장안으로 채택. 이유: (1) 버그 간 의존 구분 불가 → 한 번에 고치면 어느 수정이 효과냈는지 인과 격리 불가, (2) 사용자의 "차근차근" 원칙과 정합, (3) revert는 Git 이력 보존되므로 이전 코드 참조 가능, (4) 각 재도입 단계마다 실측하면 "기능↔회귀" 1:1 매핑 가능. 교훈: **여러 기능이 누적되며 동시다발적 회귀가 발생한 경우, "어디까지 고장났는지" 보다 "어디부터 다시 쌓을지" 관점이 더 효율적**. 앞으로 회귀 3개 이상이 얽힌 상태에서는 hotfix 대신 revert+재도입을 기본 옵션으로 고려한다.

### [2026-04-08] show_pdf_page는 CMYK를 "물리적으로" 보존하지만 "메타데이터 관점"에서는 숨길 수 있다

- **발견자**: tester (6단계)
- **내용**: PyMuPDF `show_pdf_page`는 원본 페이지를 Form XObject로 캡슐화해 새 페이지에 Do 연산자로 호출한다. 이 방식은 원본의 벡터/색상 정보를 있는 그대로 유지하므로 **인쇄 결과물은 정확히 CMYK**다. 하지만 상위 페이지 콘텐츠 스트림에서는 색상 연산자가 직접 보이지 않으므로, 콘텐츠 스트림만 검사하는 도구는 "색상 공간 불명"으로 오판한다. PDF 색상 분석 도구를 만들 때는 항상 (1) 페이지 콘텐츠 (2) Form XObject 스트림 (3) 리소스 딕셔너리의 ColorSpace 엔트리 세 곳을 모두 확인해야 완전하다.
